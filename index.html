1. Overview: Summary & Business Outcomes

The Braze and Talon.One integration enables a seamless, bidirectional flow of data between the customer journey (Braze) and the incentive logic (Talon.One). This two-way communication architecture is critical: Braze pushes customer profile updates, segment changes, and behavioral triggers to Talon.One, while Talon.One instantly returns personalized offers, coupon codes, loyalty balances, and reward notifications back to Braze for immediate customer engagement.

Instead of static, generic discounts managed in disconnected systems, promotions become dynamic, context-aware, and deeply integrated into every touchpoint of the user experience. When a customer qualifies for a reward in Talon.One, Braze knows instantly—no batch delays, no manual data exports. When Braze identifies a high-value segment, Talon.One can immediately apply exclusive offer eligibility rules.

Why Bidirectional Integration Matters:
  • Real-Time Personalization: Customer actions in one system instantly influence the other—no waiting for overnight batch jobs
  • Closed-Loop Attribution: Track which Braze campaigns drove which Talon.One redemptions, enabling true ROI measurement
  • Contextual Intelligence: Combine Braze's engagement data (opens, clicks, channel preferences) with Talon.One's transaction data (purchase history, loyalty tier, coupon usage) for hyper-relevant incentives
  • Operational Efficiency: Marketers configure rules once in Talon.One, and Braze automatically delivers the right offer to the right customer at the right time across all channels

Key Business Outcomes

Reduced Promotion Leakage

Real-time validation ensures codes are used only by eligible customers.

Increased Conversion Rates

Deliver high-relevance incentives via the right channel at the moment of intent.

Enhanced Loyalty Engagement

Use real-time ledger updates to trigger "Milestone" or "Tier Upgraded" messages.

Operational Agility

Marketers launch campaigns via the Rule Builder without requiring dedicated engineering sprints.

3. How to Setup: Guidelines & Data Flows

To achieve a successful integration, both platforms must use a shared Integration ID (typically the Braze external_id).

Keep Talon.One attributes in sync with Braze user profiles to power rule-based targeting.

Data Flow: Profile Sync

↓

Braze

↓
3

Braze sends PUT request to Talon.One Customer Profile API with updated attributes
↓
4

↓

Setup Steps:
  • Create a Braze Webhook Campaign triggered by an attribute change or sign-up event
  • Configure endpoint: PUT https://[your_deployment].talon.one/v1/customer_profiles/{{${user_id}}}

4. Guidelines for Braze Dashboard Configuration

Proper configuration in the Braze dashboard ensures secure, reliable integration with Talon.One. Follow these best practices to avoid common pitfalls and ensure smooth operation.

Manage Settings: API Key Security

Ensure your Talon.One API Key is stored in Global Settings or Content Blocks in Braze for security. This prevents hardcoding sensitive credentials directly in individual campaigns.

Best Practice:

  • Navigate to Settings → Manage Settings → Connected Content
  • Add your Talon.One API key as a credential with a descriptive name (e.g., talon_api_key)
  • Reference it in webhooks using Liquid: {{connected_content.talon_api_key}}
  • This approach enables centralized key rotation without updating individual campaigns

IP Whitelisting: Network Access Control

If your Talon.One deployment is behind a firewall, whitelist Braze's Production IP ranges to ensure Webhooks and Connected Content calls are not blocked.

Configuration Steps:

  • Obtain Braze's current IP ranges from their documentation (varies by instance: US-01, US-02, EU-01, etc.)
  • Add these IP ranges to your Talon.One deployment's firewall allowlist
  • Test connectivity after whitelisting to confirm requests are no longer blocked
  • Set up monitoring alerts for any failed connection attempts from Braze IPs

Testing: Validate Before Launch

Always use the "Test" tab in the Braze Webhook composer to verify that Talon.One returns a 201 Created or 200 OK response before activating a Canvas.

Testing Checklist:

  • Webhook Test: Send a test payload with sample user data and verify the response status code and body
  • Connected Content Test: Preview an email/push with Connected Content tags to ensure coupon codes or loyalty balances are fetched correctly
  • Error Handling: Test failure scenarios (invalid API key, non-existent user) to confirm graceful degradation
  • Rate Limit Configuration: Set appropriate rate limits for your campaigns to prevent overwhelming the API during large sends
  • Canvas Test User: Run a full Canvas journey with a test user profile to validate end-to-end integration before production launch
Pro Tip: Create a dedicated "Talon.One Integration Testing" segment in Braze with internal team email addresses. Use this segment for all integration testing to avoid impacting real customers during configuration and troubleshooting.

5. Infrastructure: Performance & Reliability

⚡ Critical Performance Requirement: Connected Content Timeout

Braze's Connected Content has a 2-second timeout for API calls. If the external API (like Talon.One) doesn't respond within 2 seconds, the Connected Content call fails and the message sends without the dynamic content—resulting in emails with missing coupon codes or broken personalization.

Talon.One's sub-100ms latency is purpose-built for this requirement. With response times 20x faster than Braze's timeout threshold, Connected Content calls succeed reliably even during peak traffic, ensuring every email includes the personalized offer or loyalty balance without risk of timeouts.

🔑 Simple API Key Integration vs Complex Tokenization

Talon.One uses API key-based authentication, where a single API key provides access to multiple endpoints (customer profiles, coupons, loyalty ledger, audiences, campaigns). This simplifies integration: configure the API key once in Braze's Global Settings, then reference it across all webhooks and Connected Content calls.

Legacy platforms often use OAuth tokenization, requiring:

  • Multi-step token exchange flows (client_id, client_secret, authorization endpoints)
  • Token expiration and refresh logic (tokens expire every 1-24 hours, requiring automated refresh)
  • Separate tokens for different API scopes or environments
  • Complex error handling for expired/invalid tokens mid-campaign

Result: Talon.One's API key approach enables marketers to configure integrations directly in Braze without engineering support. Legacy tokenization systems require dedicated backend services to manage token lifecycle, adding operational complexity and points of failure.

Sub-100ms Latency

Talon.One is engineered for high-velocity environments with a globally distributed infrastructure that ensures lightning-fast response times. This ensures that when Braze makes a Connected Content call to fetch a coupon during a message send, the response is near-instant, preventing timeouts and ensuring a smooth user experience.

Technical Details:
  • Global Edge Network: API requests are routed to the nearest data center, minimizing network latency across geographies
  • In-Memory Caching: Frequently accessed data (customer profiles, campaign rules) are cached in-memory for instant retrieval
  • Optimized Rule Engine: Rule evaluation happens in microseconds, even with complex conditional logic and multiple data points
  • Real-World Impact: During peak promotional periods (Black Friday, flash sales), sub-100ms latency ensures millions of concurrent coupon requests are processed without degradation

Resilient Webhook Retry Logic

Talon.One implements industry-leading webhook reliability to ensure no promotional event is lost due to temporary network issues or downstream system failures.

Retry Mechanism Details:
  • Retry Policy: Talon.One attempts to resend a failed webhook up to 10 times or until a 2xx success response is received. Each attempt is logged for full audit trail visibility.
  • Exponential Backoff Strategy: Initial retry after 5 minutes, then exponentially increasing intervals (10m, 20m, 40m, 80m, etc.). This prevents overwhelming the receiving system during recovery.

⚠️ What Happens When Webhooks Fail (Enterprise Marketing Platforms vs Talon.One)

Scenario: A customer completes a $200 purchase. The promotion system awards 200 loyalty points and attempts to send a webhook to Braze to trigger a "Points Earned!" push notification.

❌ Enterprise Marketing Platforms (Limited Retry Windows):

  • Retry Configuration: 5 attempts maximum with 24-hour TTL, completing all attempts in ~3 minutes
  • The Problem: Systems recover in 10-30 minutes, but retries (1s, 2s, 4s, 8s) exhaust before recovery completes
  • Result: System gives up before Braze comes back online. Customer earns points but receives no notification. Systems fall out of sync
  • Business Impact: 3-5% silent failure rate during incidents. Thousands of missed notifications. Loyalty programs appear broken to customers

✅ Talon.One (Extended Retry Logic - 10 Attempts Over Hours):

  • Retry Configuration: 10 attempts with exponential backoff (5m, 10m, 20m, 40m, 80m intervals). No TTL cap
  • The Advantage: Patient retry strategy gives systems adequate recovery time. Succeeds on attempt #4 (35 minutes total)
  • Result: Customer receives notification within 35 minutes. Data consistency maintained across both systems. No manual intervention needed
  • Business Impact: 99.9%+ automatic recovery. Near-zero silent failures. Marketing campaigns deliver expected ROI even during infrastructure incidents

💡 Key Takeaway: Enterprise platforms inherit retry policies from email systems (optimized for seconds), not cross-system integration where 10-30 minute recovery is normal. Talon.One's approach matches distributed systems reality.
